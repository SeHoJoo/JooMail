# JooMail 웹메일 UI 기획 문서

> 이 문서는 **기획 문서**입니다. 코드는 포함하지 않습니다.
> 목적: Dovecot/Postfix 백엔드 위에서 동작할 브라우저 기반 웹메일 클라이언트의 화면 구조와 UI 설계 방향을 확정하여, 다음 개발 세션이 바로 구현에 들어갈 수 있도록 하는 것.
>
> **개정 (2026-07-02)**: 사용자 리뷰 피드백 반영 — MIME 인코딩 파싱 요구사항 명시, 기능 최소화 원칙 및 제외 기능 목록 추가, 멀티 계정 지원을 확정 사항으로 반영, 프론트엔드 스택을 Vite + React + TypeScript로, 백엔드 스택을 Go로 변경.
> **개정 (2026-07-02)**: 현재 개발 범위를 실제 IMAP/SMTP 데이터 기반으로 변경. 제품 동작에서 mock account/mail data를 사용하지 않는다.

이 문서는 두 부분으로 나뉩니다.

- **Part A — 제품/UX 기획**: 무엇을 만들지, 어떻게 동작해야 하는지에 대한 결정사항. 구현 방법은 다루지 않는다.
- **Part B — 구현 권장안**: Part A를 실제 코드로 옮기기 위한 스택, 컴포넌트 분해, 단계별 작업 순서. 여기서도 코드 작성은 하지 않으며, "다음에 무엇을 어떤 순서로 만들지"만 정의한다.

범위 밖(이 문서에서 결정하지 않음): 데이터베이스, 서버 배포, Dovecot/Postfix 구성, 장기 계정 자격증명 저장 방식, 백그라운드 mailbox sync/indexing 방식.

---

## Part A — 제품/UX 기획

### 1. 제품 목표 (Product Goal)

JooMail은 이미 운영 중인 Dovecot/Postfix 메일 서버를 위한 **실용적인 브라우저 웹메일 클라이언트**다. 목표는 다음과 같다.

- 자체 호스팅 메일 서버를 가진 기술 사용자(개인 또는 소규모 팀)가 브라우저에서 **여러 메일 계정**을 오가며 빠르게 확인·정리·작성할 수 있게 한다.
- Roundcube류의 오래된 웹메일 UI 대비 **더 빠르고, 더 조밀하고, 더 조작 중심적인** 경험을 제공한다.
- 마케팅 랜딩 페이지처럼 보이지 않는, **도구(tool)로서의 UI**를 지향한다. 큰 일러스트, 프로모션 배너, 장식적 애니메이션은 없다.
- **기능은 필수 기능부터 구현한다.** 대화 스레딩, 규칙/필터, 예약 발송/발송 취소는 2026-07-03 제품 결정으로 이후 phase 범위에 포함하되, 저장소/작업 큐가 필요한 기능은 별도 설계를 거쳐 구현한다. 라벨/태그 시스템은 현재 규칙 단계에 포함하지 않는다.
- 현재 마일스톤은 실제 IMAP/SMTP 데이터 기반의 웹메일 동작을 다룬다. 제품 화면은 mock account/mail data를 사용하지 않는다.

성공 기준(정성적):
- 받은편지함을 열었을 때 한 화면에서 최대한 많은 메일을 스캔할 수 있다.
- 키보드만으로 읽기 → 답장 → 다음 메일 이동이 가능하다.
- 계정 전환이 2클릭 이내로 끝난다.
- 어떤 인코딩/형식의 메일이 와도 본문이 깨지지 않고 정확히 표시된다.
- 로딩/에러/빈 상태가 메일 앱을 써본 사용자에게 낯설지 않다(관습을 따른다).

### 2. 주요 사용자 워크플로우 (Primary User Workflows)

우선순위 순서대로:

1. **훑어보기(Triage)** — 받은편지함을 열고 제목/발신자/스니펫만 보고 중요한 것과 아닌 것을 빠르게 구분한다. 대부분의 세션은 이 동작으로 시작한다.
2. **읽기** — 메일 하나를 열어 본문과 첨부파일을 확인한다. 본문은 원본 MIME 인코딩과 무관하게 정확히 파싱되어 보여야 한다(9번 항목 참고).
3. **답장 / 전체답장 / 전달** — 읽던 메일에 곧바로 응답한다. 컨텍스트(원본 리스트 위치)를 잃지 않아야 한다.
4. **새 메일 작성** — 어느 화면에서든 접근 가능해야 하며, 다계정 환경에서는 보내는 계정을 선택할 수 있어야 한다.
5. **검색** — 발신자, 제목, 본문 키워드로 전체 메일함 또는 특정 폴더를 검색한다.
6. **정리** — 읽음/안읽음 전환, 보관, 삭제, 폴더 이동, 플래그(중요 표시)를 한다. 단건 및 다건(체크박스 다중 선택) 모두 지원한다.
7. **계정 전환** — 사이드바에서 등록된 계정 중 하나로 전환하면 해당 계정의 폴더 트리와 메일함으로 전체 화면이 갱신된다.
8. **폴더 탐색** — 현재 선택된 계정 내에서 다른 메일함(Sent, Drafts, Archive, 사용자 정의 폴더)으로 이동한다.

워크플로우 1과 6은 사용 빈도가 가장 높으므로, 리스트 뷰의 상호작용 비용(클릭 수, 시선 이동)을 최소화하는 것이 설계의 핵심 기준이다. 워크플로우 7(계정 전환)은 빈도는 낮지만 실패 시 사용자가 "지금 어느 계정을 보고 있는지" 혼란을 겪으므로 시각적 명확성이 중요하다.

### 3. 정보 구조 (Information Architecture)

```
Accounts (다중 IMAP 계정, 사용자가 추가하고 전환 가능)
 └─ Account (예: jooseho@gmail.com, work@company.com — 각각 독립된 IMAP 연결)
     └─ Mailbox (해당 계정의 IMAP 폴더 트리)
         ├─ Inbox
         ├─ Starred / Flagged (가상 폴더, 시스템 제공)
         ├─ Sent
         ├─ Drafts
         ├─ Archive
         ├─ Spam
         ├─ Trash
         └─ 사용자 정의 폴더 (중첩 가능, 예: Work/Clients, Work/Internal)
             └─ Message (메일 1건 — MIME 파싱된 헤더/본문/첨부 구조)
                 └─ Attachment (0개 이상)
```

- 계정은 서로 독립적이다. **통합 받은편지함(unified inbox)은 MVP에서 제공하지 않는다** — 기능 최소화 원칙(12번 항목)에 따라, 항상 "현재 선택된 계정" 하나의 폴더 트리만 화면에 보인다.
- 각 계정은 자신만의 폴더 트리, 리스트 상태(선택/스크롤 위치), 읽기창 상태를 독립적으로 유지한다(계정을 전환했다가 되돌아오면 이전 상태가 복원되는 것이 이상적이나, 이는 1단계 필수는 아니다).

라우팅 구조(제안, 실제 프레임워크 결정은 Part B):

- `/mail/:accountId/:mailboxId` — 메시지 리스트 + (있다면) 선택된 메시지
- `/mail/:accountId/:mailboxId/:messageId` — 특정 메시지가 열린 상태
- `/search?accountId=...&q=...&scope=:mailboxId|account` — 검색 결과 화면 (계정 간 통합 검색은 하지 않음 — 기능 최소화 원칙)
- `/compose` — 신규 작성(모바일에서는 전체 화면 라우트, 데스크톱에서는 오버레이이므로 라우트 없이 상태로만 존재 가능). 기본 발신 계정은 현재 선택된 계정.

### 4. 주요 화면 (Main Screens)

| 화면 | 설명 | 현재 범위 |
|---|---|---|
| 로그인 화면 | IMAP 계정 자격 증명 입력 및 세션 시작 | 포함 |
| 계정 추가/전환 | 사이드바 상단의 계정 스위처(드롭다운 또는 리스트) + 계정 추가 폼 | 포함 |
| 메인 메일 화면 | 사이드바 + 리스트 + 읽기창 3단 구성 (현재 선택된 계정 기준) | 포함 |
| 검색 결과 화면 | 메인 화면의 리스트 영역이 검색 모드로 전환된 형태 | 포함 |
| 작성/답장 화면 | 오버레이(데스크톱) 또는 전체화면(모바일), 발신 계정 선택 및 SMTP 전송 포함 | 포함 |
| 설정 화면 | 표시 밀도 등 최소한의 UI 환경설정 | 자리만 확보, 상세 설계는 후순위 |

로그인/계정 추가 화면은 실제 IMAP LOGIN 검증을 통해 세션을 시작한다. 별도 사용자 DB나 자체 비밀번호 시스템은 도입하지 않는다.

### 5. 데스크톱 레이아웃 (Desktop Layout)

3단 컬럼(3-pane) 레이아웃을 기본으로 한다. Gmail/Superhuman/Fastmail과 동일한 관습이며, 학습 비용이 낮다.

```
┌─────────────────────────────────────────────────────────────────┐
│ TopBar: [☰] [검색............................] [작성] [계정 ▾]    │
├───────────────┬───────────────────────┬───────────────────────────┤
│ Sidebar       │ Message List          │ Reading Pane              │
│ ┌───────────┐ │ (~360–420px,          │ (flex, 나머지 전체)         │
│ │계정 스위처│ │  고정폭 또는 리사이즈) │                            │
│ │jooseho@.. ▾│ │                       │  제목                      │
│ └───────────┘ │ ☐ 발신자  제목  ...   │  발신자 → 수신자            │
│ Inbox   12    │ ☐ 발신자  제목  ...   │  [답장][전체답장][전달][⋯]  │
│ Starred       │ ☐ 발신자  제목  ...   │  ─────────────────────    │
│ Sent          │ ...                   │  (MIME 파싱된) 본문...      │
│ Drafts   2    │                       │                            │
│ Archive       │                       │  첨부파일: file.pdf (2MB) │
│ Trash         │                       │                            │
│ ── Folders ── │                       │                            │
│ Work          │                       │                            │
│  └ Clients    │                       │                            │
└───────────────┴───────────────────────┴───────────────────────────┘
```

- 사이드바 최상단에 **계정 스위처**를 고정 배치한다. 현재 계정 이메일 주소를 표시하고, 클릭 시 등록된 계정 목록 + "계정 추가" 항목이 드롭다운으로 열린다.
- 사이드바는 접을 수 있어야 한다(아이콘만 남는 축소 모드). 리스트와 읽기창의 폭 비율은 사용자가 드래그로 조절 가능해야 한다(리사이즈 핸들).
- 리스트와 읽기창 사이의 폭 비율은 로컬 저장(localStorage 등)으로 유지한다 — 세션마다 초기화되면 안 된다.
- 최소 지원 뷰포트: 1280px. 그보다 좁아지면 리스트 폭을 자동으로 줄이되, 읽기창이 먼저 사라지지 않도록 한다(태블릿 대응은 6번 항목의 규칙을 따른다).

### 6. 모바일 레이아웃 (Mobile Layout)

3단 컬럼을 동시에 보여줄 공간이 없으므로, **드릴다운(drill-down) 방식의 단일 컬럼 네비게이션**을 사용한다.

```
[사이드바 화면]        [리스트 화면]           [읽기 화면]
┌─────────────┐      ┌─────────────┐       ┌─────────────┐
│ jooseho@.. ▾│      │ ← Inbox   🔍│       │ ← 제목       │
│             │  →   │             │   →   │  발신자      │
│ Inbox   12  │      │ 메일 1      │       │             │
│ Starred     │      │ 메일 2      │       │  본문...     │
│ Sent        │      │ 메일 3      │       │             │
│ ...         │      │ ...         │       │ [답장][⋯]   │
│         [+] │      │             │       │             │
└─────────────┘      └─────────────┘       └─────────────┘
```

- 화면 전환은 좌→우 네비게이션 스택으로 처리한다(뒤로가기 = 이전 화면).
- 상단 바에는 항상 뒤로가기와 현재 컨텍스트(폴더명 또는 메일 제목)를 표시한다. 사이드바 화면의 최상단은 데스크톱과 동일하게 계정 스위처다.
- 작성(Compose)은 전체화면 모달로 연다. 최소화 개념은 모바일에는 없다(닫거나 임시저장만 가능).
- 리스트에서의 스와이프 제스처는 **1단계에 포함하지 않는다**(기능 최소화 원칙, 12번 항목). 각 리스트 항목에 탭 가능한 오버플로 메뉴(⋯)로 보관/삭제 등을 제공한다.
- 브레이크포인트 기준(제안): `< 768px`을 모바일 레이아웃으로 취급, `768–1279px`은 사이드바를 오버레이(버튼으로 열고 닫는 방식)로 처리한 2단 컬럼(리스트+읽기창)으로 취급한다.

### 7. 메일함/사이드바 구조 (Mailbox / Sidebar Structure)

- **계정 스위처**: 사이드바 최상단 고정. 현재 계정 표시 + 드롭다운(등록된 계정 목록, 각 항목에 안읽음 총합 카운트, 하단에 "계정 추가" 항목). 계정을 선택하면 그 아래 폴더 트리 전체가 해당 계정 기준으로 교체된다.
- **시스템 폴더** (고정 순서, 항상 상단에 노출): Inbox, Starred(가상 폴더 — 플래그된 메일만 모아보기), Sent, Drafts, Archive, Spam, Trash.
- **사용자 정의 폴더**: IMAP 서버가 보고하는 폴더 트리를 그대로 반영한다. 중첩 폴더는 들여쓰기 + 접기/펼치기(chevron)로 표현한다.
- 각 폴더 항목 구성: `[아이콘] 폴더명 [안읽음 카운트]`. 안읽음 카운트는 숫자만 표시하고(배지 배경색 최소화), 0이면 표시하지 않는다. Sent/Drafts처럼 안읽음 개념이 없는 폴더는 카운트를 표시하지 않는다.
- 현재 선택된 폴더는 배경색 하나로만 구분한다(굵은 테두리나 그림자 없이 — "calm" 원칙).
- 작성(Compose) 버튼은 계정 스위처 바로 아래, 사이드바 상단에 고정 배치한다(리스트를 스크롤해도 항상 보여야 함).

### 8. 메시지 리스트 동작 (Message List Behavior)

한 행(row)에 표시할 정보(좌→우):

`[☐ 체크박스] [읽음 상태 dot] [발신자] [제목 — 스니펫] [📎 첨부 여부] [★ 플래그] [날짜/시간]`

- **밀도**: 컴팩트 모드 고정(행 높이 약 40–44px, 폰트 13–14px). "편안한(comfortable)" 모드 같은 표시 밀도 선택 옵션은 **1단계에서 제공하지 않는다** — 설정 항목 자체를 늘리지 않는 것이 기능 최소화 원칙에 부합한다.
- **정렬**: 최신순(날짜 내림차순) 고정. 발신자순/안읽음 우선 등 정렬 옵션은 제공하지 않는다(제외 기능, 12번 항목).
- **선택 동작**:
  - 단일 클릭 → 해당 메일을 읽기창에 연다(리스트는 유지).
  - 체크박스 클릭 → 다중 선택 모드 진입, 상단에 "N개 선택됨" 액션 바가 나타난다(보관/삭제/이동/읽음처리 일괄 버튼).
  - Shift+클릭으로 범위 선택, Cmd/Ctrl+클릭으로 개별 추가 선택을 지원한다(데스크톱).
- **로딩 방식**: 가상 스크롤(virtualized) 기반 무한 스크롤을 권장한다. 전통적 페이지네이션(1/2/3페이지 버튼)은 사용하지 않는다 — 훑어보기 워크플로우와 맞지 않는다.
- **호버 액션**: 마우스 호버 시 행 우측에 보관/삭제/읽음전환 아이콘이 나타난다(항상 보이지 않고 호버 시에만 — 리스트를 조용하게 유지).
- **스레딩(대화 묶기)**: 이후 phase 범위에 포함한다. 1차 구현은 `Message-ID`, `In-Reply-To`, `References` 기반 스레드 메타데이터를 백엔드가 파싱하고, 이후 리스트 묶기/스레드 보기 UX를 추가한다.

### 9. 읽기 창 동작 (Reading Pane Behavior)

- **헤더 영역**: 제목(굵게, 리스트보다 한 단계 큰 크기) → 발신자 이름/주소(기본은 이름만, 클릭 시 전체 주소 펼침) → 수신자/참조는 기본 접힘, "받는사람 보기" 토글로 펼침 → 날짜/시간(전체 타임스탬프, 호버 시 상대시간과 절대시간 모두 확인 가능).
- **액션 바**: 답장 / 전체답장 / 전달을 1차 버튼으로, 보관·삭제·이동·플래그·안읽음표시·더보기(⋯)는 2차 버튼 또는 오버플로 메뉴로 배치한다.
- **첨부파일**: 헤더 하단에 파일명·크기·타입 아이콘·다운로드 버튼 목록으로 표시. 이미지 첨부는 썸네일을 제공한다.
- **인용/이전 답장 스레드**: 본문 내 인용된 이전 메일(`>` 인용구, Gmail 스타일 `On ... wrote:` 블록)은 기본 접어서(collapsed) 보여주고 클릭 시 펼친다.
- **선택된 메일이 없을 때**: 빈 상태를 보여준다(11번 항목 참고). 리스트만 있고 읽기창이 비어있는 상태는 초기 진입 시 정상적인 상태다.

#### 9.1 MIME 본문 파싱 및 인코딩 처리 (필수 요구사항)

실제 메일은 단순 텍스트가 아니라 MIME 구조로 인코딩되어 있으므로, 읽기창은 원문을 있는 그대로 보여주는 것이 아니라 **파싱된 결과**를 보여줘야 한다. 다음 항목은 화면에 도달하기 전에 반드시 처리되어야 하는 것들이다.

- **Content-Type / multipart 구조 해석**
  - `multipart/alternative`: `text/html`과 `text/plain`이 함께 오면 기본적으로 `text/html`을 렌더링하고, HTML 파트가 없으면 `text/plain`을 사용한다(줄바꿈 보존, URL 자동 링크화 적용).
  - `multipart/mixed`: 본문과 첨부파일이 섞여 온 경우, 첨부는 첨부 목록으로 분리하고 본문만 읽기 영역에 렌더링한다.
  - `multipart/related`: 본문 HTML이 인라인 이미지를 참조하는 경우(`<img src="cid:...">`), `Content-ID`와 실제 첨부 파트를 매핑해 이미지가 깨지지 않고 보이도록 처리한다.
- **Content-Transfer-Encoding 디코딩**: `base64`, `quoted-printable`, `7bit`/`8bit` 등 인코딩 방식에 따라 원문을 디코딩한 후 렌더링한다.
- **문자셋(charset) 변환**: 메일 헤더에 선언된 charset(예: `UTF-8`, `EUC-KR`, `ISO-2022-JP` 등— 자체 호스팅 환경에서는 오래된 발신 클라이언트로부터 비 UTF-8 인코딩 메일이 실제로 유입될 수 있다)을 UTF-8로 변환한 뒤 화면에 표시한다. 변환에 실패하면 "이 메일의 인코딩을 확인할 수 없습니다" 같은 조용한 안내와 함께 가능한 한 원문을 그대로 보여준다(깨진 화면을 빈 화면보다 우선).
- **HTML sanitize**: 파싱된 HTML은 반드시 sanitize 후 렌더링한다(스크립트/이벤트 핸들러 제거). 원격 이미지는 기본 차단하고 "이미지 표시" 확인 버튼을 노출한다(프라이버시/보안 기본값).
- **책임 분리**: 이 파싱 작업은 프론트엔드가 아니라 **백엔드(Go)의 책임**이다. 프론트엔드는 이미 디코딩·구조화된 결과(예: `{ textBody, htmlBody, attachments: [...], headers: {...} }` 형태의 데이터)를 받아 렌더링만 담당한다. 프론트엔드에서 raw MIME 문자열을 직접 파싱하지 않는다 — 이는 Part B의 스택 결정(13번 항목)과 직결된다.
- **파싱 실패 시**: 본문을 표시할 수 없는 경우(손상된 메일, 지원하지 않는 인코딩 등) 에러 상태(11번 항목)를 따르며, 최소한 원본 헤더(제목/발신자/날짜)는 항상 보여준다.

### 10. Compose 플로우 (Compose Flow)

- **진입점**: (a) 사이드바 상단 "작성" 버튼, (b) 리스트/읽기창의 답장·전체답장·전달 버튼.
- **발신 계정 선택**: compose 상단에 "보내는 사람" 드롭다운을 두어, 등록된 계정 중 하나를 선택할 수 있게 한다. 기본값은 현재 화면에서 보고 있던 계정(답장/전체답장/전달의 경우 원본 메일이 수신된 계정)이다.
- **데스크톱 형태**: 화면 우측 하단에 고정된 오버레이 카드(Gmail 스타일). 리스트/읽기창을 가리지 않고 동시에 볼 수 있어야 한다 — "메일을 보면서 답장을 쓰는" 워크플로우를 지원하기 위함이다.
  - 오버레이는 최소화(제목 표시줄만 남기기), 전체화면 확장, 닫기(임시저장 확인)를 지원한다.
  - 1단계는 **동시에 하나의 compose 창**만 지원한다(Gmail처럼 여러 개 동시 최소화하는 기능은 제외 기능, 12번 항목).
- **모바일 형태**: 전체화면 모달. 뒤로가기 시 임시저장 여부를 확인한다.
- **필드 구성**: 보내는 사람(계정 선택) — 받는사람(To) — 참조/숨은참조는 기본 숨김, "참조/숨은참조 추가" 링크로 노출 — 제목 — 본문(최소한의 서식: 굵게/기울임/링크/목록만, 과도한 리치 에디터 지양) — 첨부파일 추가(드래그 앤 드롭 + 버튼).
- **상태 표시**: "임시저장됨" 텍스트를 조용히 표시한다(저장 로직 자체는 이후 마일스톤이며, 1단계는 상태 텍스트 UI만 존재하면 된다).
- **하단 액션**: 보내기(주 버튼), 임시보관함으로 저장 후 닫기, 삭제(휴지통 아이콘).

### 11. 필수 UI 상태 (Required UI States)

모든 상태는 "조용한" 톤을 유지한다 — 밝은 색 배너, 큰 아이콘, 격려성 문구는 지양한다.

| 상태 | 적용 위치 | 규칙 |
|---|---|---|
| **Empty** | 빈 편지함, 선택된 메일 없음, 검색 결과 0건 | 중앙에 짧은 문구 1줄 + 보조 설명 1줄. 일러스트 없음. 예: "받은편지함이 비어 있습니다." |
| **Loading** | 리스트/읽기창 최초 로딩, 폴더 전환, 계정 전환 | 스켈레톤 행(리스트) / 스켈레톤 블록(읽기창). 스피너는 최소화하고 스켈레톤을 우선한다. |
| **Error** | 네트워크 실패, 폴더 로딩 실패, 본문 파싱 실패 | 인라인 오류 메시지 + "다시 시도" 버튼. 전체 화면을 덮는 에러 페이지는 지양(리스트/읽기창 영역 내 국소 표시). |
| **Selected** | 리스트 항목 클릭/다중 선택 | 단일 선택: 배경색 하이라이트. 다중 선택: 체크된 행 강조 + 상단 액션 바 노출. |
| **Unread** | 읽지 않은 메일 | 발신자/제목 굵게(bold) + 좌측 dot 표시. 읽으면 즉시 폰트 굵기와 dot이 사라진다(애니메이션 없이 즉시). |
| **Search results** | 검색 실행 후 | 리스트 상단에 "'키워드'에 대한 결과 N건, [범위: 현재폴더/현재계정 전체 ▾]" 표시. 매칭된 키워드는 제목/스니펫 내 하이라이트 처리. 결과 0건이면 Empty 상태 규칙을 따르되 "검색어를 확인해주세요" 문구 사용. |

### 12. 디자인 방향 (Design Direction)

원칙: **밀도 있게(Dense), 차분하게(Calm), 도구답게(Operational), 최소하게(Minimal)**. 마케팅 페이지가 아니다.

- **레이아웃**: 여백은 기능적으로만 사용한다(구분을 위한 최소 여백). 히어로 섹션, 큰 타이포 헤드라인, 곡선형 카드, 그림자가 많은 UI는 사용하지 않는다.
- **컬러**: 뉴트럴 그레이 스케일을 기본으로 하고, 액센트 컬러는 1개만 사용한다(선택 상태, 주 액션 버튼, 링크에만 적용). 안읽음/에러 등 의미 전달용 색상(semantic color)은 최소한으로 — 빨강(에러/삭제), 액센트색(선택/주 액션) 정도로 제한한다.
- **타이포그래피**: 시스템 폰트 스택(플랫폼 기본 sans-serif) 사용. 기본 본문 13–14px, 리스트 행은 그보다 살짝 작거나 같게.
- **모션**: 트랜지션은 즉각적이고 짧게(100–150ms 이하). 바운스, 스프링 애니메이션, 진입 애니메이션은 사용하지 않는다.
- **아이콘**: 하나의 아이콘 세트로 통일(예: 선 굵기 1.5–2px의 outline 스타일 아이콘 세트).

#### 12.1 기능 최소화 원칙 (필수 요구사항)

JooMail 1단계는 **필수 기능만** 구현한다. 화면/컴포넌트를 설계할 때 "있으면 좋은" 기능을 추가하지 않는다. 판단 기준: 위 2번 항목(주요 워크플로우 1–8)을 수행하는 데 반드시 필요한가? 아니면 제외한다.

**포함(필수 기능)**
- 계정 추가/전환(멀티 계정), 폴더 탐색, 메일 목록 조회, 메일 읽기(MIME 파싱 포함), 답장/전체답장/전달, 새 메일 작성, 첨부파일 보기/추가, 읽음/안읽음, 보관/삭제/이동, 플래그(중요 표시), 단일/다중 선택, 검색(현재 폴더 또는 현재 계정 범위).

**명시적으로 제외(1단계뿐 아니라 별도 논의 전까지 전면 제외)**
- 통합 받은편지함(unified inbox) — 계정은 항상 개별적으로 본다.
- 스누즈(snooze).
- 연락처/주소록, 캘린더 연동.
- 계정 간 통합 검색.
- 리스트 정렬 옵션(발신자순 등), 표시 밀도 옵션(comfortable 모드).
- 모바일 스와이프 제스처.
- 리치 텍스트 고급 서식(폰트/색상/표 삽입 등) — 굵게/기울임/링크/목록만 지원.
- 다중 compose 창 동시 최소화.
- 다크모드 전환 UI(토큰 설계는 해두되 스위치는 후순위).

이 목록은 "나중에 언젠가"가 아니라 **명시적 비목표(non-goal)**다. 재도입하려면 별도 논의를 거쳐야 한다.

**이후 phase 포함(2026-07-03 결정)**
- 대화 스레딩(thread view) — `Message-ID`, `In-Reply-To`, `References` 기반 묶어보기.
- 규칙/필터 — ManageSieve로 계정별 Sieve 규칙을 관리하되, JooMail이 Dovecot/Postfix/Sieve 파일이나 서버 설정을 직접 수정하지 않는다. 현재 범위는 발신자/제목 기반 폴더 분류와 Spam/Trash 이동이며, 라벨/태그 시스템과 파괴적인 discard/block 동작은 별도 결정 전까지 제외한다.
- 예약 발송과 발송취소(undo send) — 백엔드 큐/저장소와 브라우저 내 지연 전송의 책임 경계를 먼저 결정한 뒤 구현한다.

- **다크 모드**: 색상 토큰은 처음부터 라이트/다크 두 세트로 설계해두는 것을 권장하되(13번 항목), 실제 전환 UI는 위 제외 목록에 따라 1단계에는 넣지 않는다.
- **레퍼런스가 될 만한 참고 관습**(그대로 베끼는 것이 아니라 톤 참고용): Fastmail, Superhuman, Mutt/Neomutt 같은 키보드 중심 클라이언트의 정보 밀도. Gmail의 3단 레이아웃과 compose 오버레이 패턴. 단, Gmail의 화려한 컬러 배지나 프로모션 탭, 라벨 시스템은 채택하지 않는다.

## Part B — 구현 권장안 (Implementation Recommendation)

> 아래는 현재 개발 기준이다. 제품 동작은 실제 IMAP/SMTP 데이터 기준으로 구현하며, mock account/mail data는 제품 플로우에 사용하지 않는다.

### 13. 권장 스택

**프론트엔드**
- **빌드 도구/프레임워크**: Vite + React + TypeScript — 가장 널리 쓰이는 조합으로, SPA(단일 페이지 앱) 구성에 적합하고 빌드/개발 서버 속도가 빠르다. 실제 데이터는 Go 백엔드가 JSON API로 제공하므로 SSR은 사용하지 않는다.
- **라우팅**: React Router (클라이언트 사이드 라우팅으로 3번 항목의 라우트 구조 구현).
- **스타일링**: Tailwind CSS — 밀도 높은 커스텀 레이아웃을 유틸리티로 빠르게 구성하기 적합하고, 디자인 토큰(컬러/스페이싱)을 config로 명시적으로 관리할 수 있다.
- **컴포넌트 베이스**: shadcn/ui (Radix 기반) — 접근성 있는 프리미티브(드롭다운, 체크박스, 다이얼로그, 리사이저블 패널 등)를 그대로 가져다 쓰고 스타일만 덧입힌다. 무거운 UI 라이브러리(Material UI 등)는 지양.
- **로컬 상태 관리**: 선택된 계정/메일, 다중 선택 집합, 사이드바 접힘 상태, compose 오버레이 상태 등은 클라이언트 상태이므로 Zustand 또는 React Context 중 가벼운 쪽으로 처리.
- **가상 스크롤**: `@tanstack/react-virtual` 등으로 메시지 리스트를 가상화한다.
- **API 데이터 계층**: 프론트엔드는 Go 백엔드 API만 통해 계정, 폴더, 메시지 목록, 메시지 본문, 첨부파일, 전송 결과를 읽고 갱신한다. 제품 플로우에서 로컬 mock account/mail data를 사용하지 않는다. 개발용 QA route는 loading/error/empty/search 같은 화면 상태를 강제로 재현할 수 있지만, 이는 시각 검증용 상태 트리거로 한정한다.

**백엔드**
- **언어**: Go.
- **역할**: Dovecot(IMAP)/Postfix(SMTP)와의 통신, **MIME 본문 파싱**(Go 표준 라이브러리 `net/mail`, `mime`, `mime/multipart`, `mime/quotedprintable` 및 charset 변환을 위한 `golang.org/x/text/encoding` 계열 활용), 여러 계정의 IMAP 연결 관리, 프론트엔드가 사용할 JSON API 제공.
- **API 계약**: 프론트엔드는 Go 백엔드가 이미 MIME을 파싱해 반환한 구조화된 JSON(`textBody`/`htmlBody`/`attachments`/`headers` 등)만 소비한다. 원문 MIME 파싱을 프론트엔드에서 다시 하지 않는다.
- **범위 제한**: Dovecot/Postfix 서버 구성, 데이터베이스, 백그라운드 mailbox sync/indexing, 장기 자격증명 저장 정책은 현재 범위 밖이다. 현재 백엔드는 표준 `net/http` 기반의 작은 API, IMAP LOGIN 기반 세션, live IMAP 조회/상태 변경, SMTP 전송, MIME 파싱과 계정별 메일 데이터 제공을 책임진다.

### 15. 개발 단계 (Phases)

각 단계는 실제 Go 백엔드 API와 live IMAP/SMTP 데이터를 기준으로 진행한다. 제품 플로우에서 mock account/mail data를 사용하지 않는다.

- **Phase 0 — 레이아웃 셸**: 디자인 토큰(컬러/스페이싱/타이포 스케일, 라이트+다크 두 세트) 정의, 데스크톱 3단 레이아웃 정적 골격, API에서 내려온 계정/폴더 구조 렌더링, 리사이즈 가능한 패널 경계.
- **Phase 1 — 메시지 리스트 + 읽기창 상호작용**: live IMAP 기반 메시지 목록 렌더링(가상 스크롤), 행 선택 → Go 백엔드가 MIME 파싱한 읽기창 데이터 표시, 읽음/안읽음 토글, 다중 선택 + 액션바, 호버 액션, 계정 전환 시 리스트/읽기창 전체 갱신.
- **Phase 2 — Compose UI + SMTP 전송**: 오버레이 열기/닫기/최소화, 발신 계정 선택, 필드 구성, 첨부 추가, SMTP 전송, Sent 폴더 반영.
- **Phase 3 — 상태 커버리지**: Empty/Loading/Error/Search-results 상태를 모든 관련 화면에 대해 구현하고 시각 검증. 개발용 QA route는 상태를 강제로 띄울 수 있지만, 제품 데이터 대체용 mock으로 사용하지 않는다. 본문 파싱 실패 케이스는 실제 파서 에러 경로와 테스트 fixture로 검증한다.
- **Phase 4 — 반응형/모바일**: 드릴다운 네비게이션, 브레이크포인트별 레이아웃 전환, 모바일 compose 전체화면 라우트.
- **Phase 5 — 백엔드 하드닝**: IMAP/SMTP 오류 처리, MIME 파싱 edge case, 첨부파일 다운로드, 원격 이미지 차단, 검색 성능, 세션 만료 처리, API 테스트 보강. 데이터베이스, 백그라운드 sync/indexing, 서버 배포는 별도 범위로 둔다.

### 16. 첫 번째 빌드 가능한 UI 마일스톤 정의

다음 개발 세션이 "완료"로 판단할 수 있는 최소 기준(Phase 0–1 범위):

**포함**
- 데스크톱 3단 레이아웃(사이드바/리스트/읽기창), 리사이즈 가능
- 사이드바 최상단 계정 스위처 + API에서 내려온 시스템 폴더/사용자 폴더 트리, 안읽음 카운트 표시
- live IMAP/API 기반 메시지 리스트(가상 스크롤), 컴팩트 밀도
- 리스트 항목 클릭 → 읽기창에 해당 메일 표시
- 읽기창은 Go 백엔드가 **이미 MIME 파싱한 본문**(plain/HTML/인라인 이미지/첨부 포함 케이스)을 정확히 렌더링
- 읽음/안읽음 상태 토글(클릭 시 즉시 시각 반영)
- 단일/다중 선택 + 다중 선택 액션바(보관/삭제/읽음처리는 IMAP 상태 변경으로 반영)
- Empty 상태(선택된 메일 없음, 빈 폴더) 최소 1종 이상 구현

**제외** (다음 마일스톤 이후, 또는 12.1번 항목에 따라 전면 비목표)
- Compose 오버레이 실제 구현(Phase 2)
- 검색 UI(Phase 3)
- 모바일 반응형(Phase 4)
- 다크모드 전환 UI, 대화 스레딩, 통합 받은편지함, 정렬/밀도 옵션(12.1번 항목의 제외 기능 목록)
- 데이터베이스, 백그라운드 mailbox sync/indexing, 배포 전부

### 17. 미결정 사항 / 다음 확인 필요 항목 (Open Questions)

이 문서에서 의도적으로 답을 유보한 항목. 구현 중 막히면 다시 논의 필요:

- 다크모드 토글 UI를 언제 도입할지(토큰 설계는 Phase 0에 포함, 스위치 자체의 도입 시점).
- 설정 화면의 실제 항목 범위(표시 밀도 설정은 제외했으므로, 그 외 무엇을 넣을지).
- 계정별 폴더/스크롤 상태를 계정 전환 후에도 유지할지 여부(1단계는 필수 아님으로 두었으나, 사용성상 필요해질 수 있음).
